# Agame Web优先与叙事核心设计

> **设计原则**：优先上线 Web 版本，以故事叙事为核心体验。其他系统（技能树、战斗、复杂经济）延后实现。

---

## 一、MVP 范围重新定义

### 1.1 核心原则

```
┌─────────────────────────────────────────────────────────┐
│  优先级金字塔（MVP 阶段）                                │
├─────────────────────────────────────────────────────────┤
│                                                         │
│    ┌─────────────┐                                     │
│    │   故事叙事   │  ← 绝对核心，不可削减                 │
│    └──────┬──────┘                                     │
│    ┌──────┴──────┐                                     │
│    │  玩家决策   │  ← 与叙事紧密结合                     │
│    └──────┬──────┘                                     │
│    ┌──────┴──────┐                                     │
│    │  Agent运转  │  ← 世界自动运转的基础                 │
│    └──────┬──────┘                                     │
│    ┌──────┴──────┐                                     │
│    │  简单资源   │  ← 支撑决策的最小经济系统              │
│    └─────────────┘                                     │
│                                                         │
│  延后实现：技能树、战斗系统、复杂领地、赛季系统           │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 1.2 MVP 边界

**必须实现（P0）**：
| 模块 | 功能点 | 说明 |
|------|--------|------|
| 叙事引擎 | 晨报生成、个人日志、事件呈现 | 核心体验，用 LLM 驱动 |
| 决策系统 | 二选一/三选一的选择面板 | 玩家与世界交互的主要方式 |
| Chronos | 基础世界平衡、阶段推进 | 每天运行一次，生成事件 |
| 国家Agent | 三国基础决策逻辑 | 每6小时运行，产生冲突和动态 |
| 状态存储 | 玩家状态、世界状态、NPC关系 | 支持决策的后果追踪 |
| Web前端 | 阅读界面 + 选择界面 | 纯文字为主，最小化UI |

**延后实现（P1/P2）**：
| 模块 | 原因 |
|------|------|
| 技能树系统 | 复杂度高，不是叙事必需 |
| 战斗系统 | 可用"决策结果"替代战斗过程 |
| 领地建设 | 简化为"资源点数" |
| 完整经济 | 简化为"金币/影响力"两种资源 |
| 赛季系统 | 上线后根据留存情况再开发 |

## 二、Web 技术架构简化

### 2.1 技术栈调整

```
原方案                        MVP方案
─────────────────────────────────────────────────────
Cocos Creator 3.x      →     纯 Web（React/Vue）
复杂动画/特效                  文字 + 简单CSS动画
微信小游戏                    先Web，后移植微信
```

### 2.2 MVP 技术栈

```
前端：
  - 框架：React + TypeScript
  - 样式：Tailwind CSS
  - 状态：Zustand / Redux Toolkit
  - 部署：Vercel / Cloudflare Pages

后端：
  - 运行时：Node.js
  - 框架：Express / Fastify
  - 数据库：PostgreSQL（世界状态）+ Redis（缓存）
  - LLM：Claude API（通过代理）
  - 部署：Railway / Fly.io
```

### 2.3 前端页面结构

```
┌─────────────────────────────────────────────────────────┐
│  [页面1] 晨报阅读页                                      │
│  ├─ 顶部：日期 + 季节                                    │
│  ├─ 主体：滚动新闻列表（四国新闻）                        │
│  └─ 底部：「我的事件」「行动」按钮                        │
├─────────────────────────────────────────────────────────┤
│  [页面2] 个人事件页                                      │
│  ├─ 待处理事件列表                                       │
│  └─ 每个事件：标题 + 简介 + 「处理」按钮                  │
├─────────────────────────────────────────────────────────┤
│  [页面3] 决策页                                          │
│  ├─ 事件描述（大段文字）                                  │
│  ├─ 2-4个选项按钮                                        │
│  └─ 选择后立即显示结果反馈                                │
├─────────────────────────────────────────────────────────┤
│  [页面4] 个人状态页                                      │
│  ├─ 属性：统治/军事/外交/商业/魔法（简化数值）            │
│  ├─ 声望：四国好感度                                     │
│  └─ 历史：过往决策记录                                    │
└─────────────────────────────────────────────────────────┘
```

## 三、叙事体验优先设计

### 3.1 叙事是第一优先级

```
设计决策检查清单：

任何功能加入前，先问自己：
□ 这个功能能让故事更精彩吗？
□ 没有这个功能，玩家还能体验完整的故事吗？
□ 这个功能会增加玩家的认知负担吗？

如果第1个答案是"否"，考虑延后或简化。
如果第3个答案是"是"，考虑砍掉。
```

### 3.2 叙事流程设计

**玩家登录流程**：
```
打开页面 → 加载动画（"世界正在运转..."）
        → 显示晨报（自动播放打字机效果）
        → 阅读完毕后显示「今日事件」提示
        → 点击进入待处理事件
        → 做出选择
        → 看到即时反馈
        → 返回晨报页，显示「已处理X件事件」
        → 可选择继续探索或离开
```

**晨报生成机制（MVP简化版）**：
```
1. Chronos 每日运行，生成 3-5 条新闻
   - 1条国际大事（涉及2+国家）
   - 1条玩家所在国新闻
   - 1条灰烬谷/边境新闻
   - 1-2条随机事件/人物动态

2. LLM 将状态变化转化为叙事文本

3. 每条新闻控制在 50-100 字

4. 重要新闻标记为「点击查看详情」
```

### 3.3 事件设计优先级

**P0 事件类型**（MVP必须）：
| 类型 | 示例 | 交互方式 |
|------|------|----------|
| 效忠选择 | 某国使节来访，邀你效忠 | 三选一：接受/拒绝/拖延 |
| 资源抉择 | 村民请求援助 vs 上缴税收 | 二选一 + 中间选项 |
| 人际冲突 | 两个NPC同时求你帮忙 | 选择帮助谁/都不帮 |
| 危机应对 | 天灾/匪患/敌军来袭 | 多选一应对策略 |

**P1 事件类型**（上线后补充）：
| 类型 | 说明 |
|------|------|
| 事件链 | 多章节连续叙事 |
| 探索事件 | 发现新地点/人物 |
| 贸易谈判 | 复杂的讨价还价 |

## 四、Agent 系统简化

### 4.1 MVP Agent 层级

```
┌─────────────────────────────────────────────────────────┐
│  Chronos（每日1次）                                      │
│  ├─ 读取世界状态                                        │
│  ├─ 检测是否进入下一阶段                                │
│  ├─ 生成 1-2 个重大事件种子                             │
│  └─ 调用 LLM 生成晨报大纲                               │
├─────────────────────────────────────────────────────────┤
│  国家Agent（每6小时1次，MVP简化为3国共用一个）           │
│  ├─ 根据当前阶段产生冲突                                │
│  ├─ 生成国家层面的事件                                  │
│  └─ 输出结构化的状态变化                                │
├─────────────────────────────────────────────────────────┤
│  NPC Agent（被动触发，MVP简化为预设模板）                │
│  ├─ 不主动运行                                          │
│  ├─ 当玩家与NPC交互时，根据模板生成对话                  │
│  └─ 关系变化记录在状态库                                │
└─────────────────────────────────────────────────────────┘
```

### 4.2 LLM 调用策略（成本优化）

```
分层模型策略：

Tier 1 - Claude Sonnet（重要叙事）
  使用场景：
  - 每日晨报生成（1次/天）
  - 重大事件生成
  - 玩家关键决策的后果描述

Tier 2 - GPT-3.5（日常交互）
  使用场景：
  - NPC 对话响应
  - 普通事件的描述生成
  - 状态变化的解释文本

Tier 3 - 模板填充（高频重复）
  使用场景：
  - 标准问候语
  - 资源增减提示
  - 日常新闻摘要
```

### 4.3 Prompt 模板（MVP）

**晨报生成 Prompt**：
```
你是一位史官，正在为《埃拉西亚晨报》撰写新闻。

世界当前状态：
- 游戏日：第{day}天
- 阶段：{phase}（和平/紧张/冲突/战争/重建）
- 苍龙帝国：{canglong_status}
- 霜狼联邦：{shuanglang_status}
- 金雀花王国：{jinque_status}
- 边境联盟：{border_status}

昨日发生的变化：
{changes}

请生成 4 条新闻，每条 50-80 字，风格像简明的历史记录。
每条新闻标明涉及国家。

输出格式：
{
  "news": [
    {"country": "苍龙", "content": "..."},
    {"country": "霜狼", "content": "..."},
    {"country": "金雀花", "content": "..."},
    {"country": "边境", "content": "..."}
  ]
}
```

## 五、内容生产规划

### 5.1 MVP 内容量

| 内容类型 | MVP数量 | 说明 |
|----------|---------|------|
| 开局剧本 | 4个 | 四国初始体验各1个 |
| 通用事件模板 | 20个 | 可复用的基础事件 |
| NPC预设对话 | 30个 | 每个关键NPC 5-10条 |
| 晨报模板 | 50个 | 不同场景的新闻模板 |
| 结局分支 | 8个 | MVP阶段的多种结局 |

### 5.2 内容制作工具

```
为了快速生成内容，建议开发：

1. 事件编辑器（Web界面）
   - 可视化编辑事件触发条件
   - 编写事件文本和选项
   - 测试事件分支

2. 晨报生成器（内部工具）
   - 输入世界状态
   - 预览生成的晨报
   - 人工调整输出

3. NPC对话测试器
   - 模拟与NPC对话
   - 测试不同关系值下的回复
```

## 六、开发里程碑

### 6.1 MVP 开发周期（8周）

```
Week 1-2: 基础架构
  - 后端API框架
  - 数据库设计
  - 基础LLM接入

Week 3-4: 核心叙事
  - 晨报生成系统
  - 事件系统框架
  - Chronos基础逻辑

Week 5-6: Web前端
  - 晨报阅读页
  - 决策交互页
  - 用户状态页

Week 7: 内容填充
  - 事件模板编写
  - NPC设定导入
  - 开局剧本完成

Week 8: 测试优化
  - 端到端测试
  - LLM Prompt调优
  - 性能优化
```

### 6.2 上线标准

```
□ 玩家可以注册/登录
□ 每天登录看到不同的晨报
□ 可以做出选择并看到后果
□ 世界状态可以持续保存
□ 单个事件链可以完整体验
□ 页面加载时间 < 3秒
□ LLM响应时间 < 5秒
```

## 八、风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|----------|
| LLM成本过高 | 无法持续运营 | 实现多层缓存，降低调用频率 |
| 叙事质量不稳定 | 用户体验差 | 增加人工审核环节，建立模板库 |
| 开发周期超出 | 错过上线窗口 | 严格按P0/P1/P2优先级，果断砍功能 |
| 玩家留存低 | 项目失败 | MVP阶段专注核心叙事，快速验证 |

## 九、设计检查清单

**每个功能设计时，确认以下问题**：

- [ ] 这个功能是叙事必需的吗？
- [ ] 这个功能能在Web上良好运行吗？
- [ ] 这个功能需要复杂的UI吗？
- [ ] 这个功能会增加LLM调用成本吗？
- [ ] 如果砍掉这个功能，核心体验还能成立吗？

**如果后三个问题中有超过一个"是"，考虑简化或延后。**
